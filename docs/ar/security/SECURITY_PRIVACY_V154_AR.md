# الأمان والخصوصية — Warqna v154

## ما تم بناؤه

- Laravel Sanctum للجلسات المحمولة.
- انتهاء صلاحية رموز الدخول بعد 30 يومًا.
- إظهار الجلسات وإلغاؤها من التطبيق.
- Rate Limiting منفصل للدخول، API، العمليات الحساسة والبلاغات.
- Request ID لكل طلب لتتبع الأخطاء.
- Security Headers.
- وضع صيانة مركزي.
- منع التطبيقات القديمة إجباريًا عبر إدارة الإصدارات.
- سجل تدقيق لعمليات المدير.
- نظام بلاغات وإدارة مراجعة/حظر.
- تصدير بيانات الحساب.
- طلب حذف الحساب مع فترة سماح قابلة للضبط.
- استعادة كلمة المرور وتأكيد البريد.
- صفحات: الخصوصية، الشروط، قواعد المجتمع، حذف الحساب، قواعد المنافسات، الدعم.
- معاملات قاعدة بيانات لسجل الاقتصاد والتحويلات والمشتريات الموجودة.
- تعطيل الحسابات المحلية في Production Build.

## إعدادات يجب تغييرها قبل النشر

- `APP_KEY`
- `ADMIN_PASSWORD`
- `DB_PASSWORD`
- بيانات SMTP
- بيانات TURN
- النطاقات الحقيقية في CORS وSanctum
- روابط الخصوصية والدعم
- جميع أسرار OAuth وAdMob والتوقيع

## الحذف بسبب عدم النشاط

الأمر موجود وقابل للضبط، لكنه يبدأ بوضع Dry Run حمايةً للبيانات:

```text
INACTIVE_ACCOUNT_PURGE_DAYS=30
INACTIVE_ACCOUNT_DRY_RUN=true
```

بعد مراجعة السجلات والسياسة القانونية يمكن تحويله إلى `false`. لا يُنصح بتفعيل الحذف الفعلي قبل إرسال تنبيهات للمستخدمين والتحقق من سياسة المشتريات والاحتفاظ القانوني.

## الحسابات التجريبية

يجب عدم إبقاء كلمات المرور التجريبية في قاعدة الإنتاج. بعد نشر الإنتاج:

1. غيّر كلمة مرور المدير.
2. احذف أو عطّل الحسابات التجريبية.
3. اجعل `ALLOW_LOCAL_DEMO_MODE=false`.
4. شغّل `warqna:production-check --strict`.

## الاقتصاد والتوكنز

- الخادم هو المرجع الوحيد للرصيد في الإنتاج.
- لا تقبل أي قيمة رصيد أو نتيجة مباراة من العميل.
- كل تحويل أو شراء يجب أن يعمل داخل DB Transaction.
- عمولة التحويل مضبوطة مركزيًا.
- مشتريات المتجر تُسجل كإيراد لمحفظة النظام/المدير وفق التصميم الحالي.

## إدارة البلاغات

البلاغات تدعم الإساءة، الغش، الإزعاج، انتحال الهوية والمحتوى غير المناسب. يستطيع المدير مراجعتها وتوثيق القرار واتخاذ إجراء حظر، مع تسجيل العملية في Audit Log.
